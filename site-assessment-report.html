1. Current Traffic Capacity

The site runs as a React Single Page Application (SPA) connected to Supabase for the database, authentication, and background functions. Hosting is via a CDN-based platform such as Netlify or Vercel.

Frontend Hosting (Netlify / Vercel Free Tier)

  • Handles approximately 100GB of bandwidth per month
  • Unlimited page requests served via global CDN
  • Scales automatically to millions of static asset requests with no configuration

Supabase (Free Tier)

  • 500MB database storage
  • 50,000 monthly active users
  • 5GB data transfer bandwidth
  • 500,000 Edge Function invocations per month
  • Comfortable for 1,000 to 5,000 monthly active users in early stage
Current Ceiling
Approximately 5,000 to 10,000 monthly active users and a few hundred concurrent users before hitting Supabase free tier limits. The frontend is not the bottleneck. Database query throughput and Edge Function cold starts will be the first constraints at scale.

2. Security Issues

Critical Issues (Must Fix Before Accepting Real Users)

Critical — Fake Payment Integration
The payment form collects card details directly in the browser and simulates a transaction with a randomly generated fake ID. No money is actually charged. The form also displays the text "Secured by Stripe • PCI DSS compliant" which is factually false. Collecting raw card numbers on the frontend violates PCI DSS compliance standards. Stripe Elements must replace the current form entirely.
Critical — Hardcoded Email Address
All outbound emails (booking confirmations, welcome emails, opportunity digests) are being delivered to a single developer email address regardless of who the intended recipient is. No users are currently receiving any emails from the platform. This is a single line fix in the Edge Function.

High Severity

High — No Rate Limiting
Any actor can repeatedly trigger booking creation, email sends, or digest runs without restriction. This exposes the platform to spam abuse, inflated email costs, and denial-of-service against the email provider quota. The digest endpoint in particular has no authentication and can be called by anyone with the URL.

Medium Severity

  • No input validation — Email addresses, phone numbers, and free-text fields are saved to the database without format checks or length limits
  • Slot bookings table allows unrestricted public INSERT and SELECT, making it trivially spammable

Low Severity / Architecture Notes

  • CORS on Edge Functions currently allows all origins. This is acceptable in development but should be locked to the production domain before launch.
  • The Supabase anonymous key is visible in the frontend bundle. This is standard and expected for SPA architecture. It is not a vulnerability on its own.

What Is Done Well

Security Strengths
  • Row Level Security (RLS) is fully implemented on all database tables with appropriately restrictive policies
  • No SQL injection vectors — all queries use the Supabase parameterised query builder
  • No hardcoded secrets in frontend code
  • No XSS vulnerabilities — React escapes all user-generated content automatically
  • CSRF is not a risk — authentication uses JWT tokens, not cookies
  • Sensitive API keys (Resend, Supabase service role) are stored only in Edge Function environment variables, never in the frontend

3. Scalability Path & Speed

The architecture scales cleanly. Supabase and Vite/Vercel are both designed for this. The database is well-indexed with GIN, trigram, and B-tree indexes already in place. The main future bottlenecks will be Edge Function cold starts and unoptimised queries as listing volume grows into the tens of thousands.

Phase 1 — Up to 50,000 Monthly Users

  • Upgrade Supabase to Pro plan
  • Database connection pooling via PgBouncer is included on Pro (no code changes)
  • No architectural changes required
  • Time to implement: same day

Phase 2 — Up to 500,000 Monthly Users

  • Add Cloudflare in front of the app for DDoS protection and edge caching
  • Implement server-side pagination on all listing queries (partially done)
  • Add Redis caching for frequently read listing data via Upstash
  • Enable read replicas in Supabase Pro/Team plan
  • Time to implement: 1 to 2 days engineering

Phase 3 — Millions of Users

  • Migrate to Supabase Team or Enterprise plan
  • Implement pg_cron for scheduled background jobs (digest emails, cleanup)
  • Split monolithic Edge Functions into smaller, purpose-specific functions
  • Consider a dedicated transactional email provider (Postmark or SendGrid) for high volume
  • Time to implement: 1 to 2 weeks engineering

6. Development Recommendations

The following features are ranked by market impact for a media slot marketplace. They are presented in order of value delivered to buyers and sellers, not technical complexity.

Immediate Priorities (Before Launch)

  1. Implement real Stripe integration — Use Stripe Elements to collect card details securely. Create a payment intent server-side via an Edge Function and confirm it client-side. Never collect raw card numbers in the frontend.
  2. Fix email routing — Remove the hardcoded email address from the Edge Function so booking confirmations, welcome emails, and notifications reach actual users.
  3. Add rate limiting — Protect the email send endpoint and booking creation from abuse using Upstash Redis or simple in-function counters.

High-Value Features for This Market

# Feature Market Impact Why It Matters
1 Seller Verification Badges Very High Verified subscriber and download counts (audited by ABC, Podtrac, or similar) command significantly higher prices and trust. A simple admin-reviewed verification request flow differentiates the platform from unverified listing sites.
2 Post-Campaign Performance Reporting Very High Sellers uploading open rate and click data after a slot runs creates a feedback loop. Buyers can filter by proven ROI. This is the primary driver of repeat purchase rates on content marketplaces.
3 Escrow Payment Flow High Hold full payment at booking and release to the seller 7 days after the slot goes live. This is the standard for content marketplaces and eliminates most buyer-seller disputes. The current deposit model is appropriate for early stage only.
4 Media Kit PDF Generation High Auto-generate a downloadable media kit from listing data. Sellers use these constantly for outreach and it drives organic traffic back to the platform when shared externally.
5 Waitlist for Sold-Out Slots High When a slot is booked, interested buyers join a waitlist. If a booking falls through, the next in line is notified automatically. Reduces revenue lost to cancellations.
6 Bulk Booking / Agency Accounts Medium Agencies booking across multiple properties need a campaign basket view and consolidated invoicing. This unlocks a higher-value buyer segment that transacts at higher volume and lower support cost.
7 RSS Feed & Zapier Webhook Medium Let buyers subscribe to new listing notifications via RSS or Zapier webhook in addition to email digest. Zapier alone has millions of users who automate procurement workflows, providing a free distribution channel.
8 Seller Analytics Dashboard Medium Show sellers pageviews per listing, click-throughs into the booking flow, and conversion rates. Sellers on a platform that shows demand data are more likely to renew and recommend the platform to peers.
9 Buyer Review System Medium Post-campaign ratings from buyers build seller trust scores over time. Essential for a marketplace to self-regulate quality and reduce the overhead of manual seller vetting.
10 Affiliate / Referral Programme Growth With a deposit model, even a 3% referral commission is sustainable. Newsletter writers who list their own slots will naturally refer other writers if there is a financial incentive. Organic acquisition at low cost.
